### 학습 내용

#### 1. 추상화(Abstraction)와 객체 간 협력 — `section03.abstraction`
- `Car`, `CarRacer` 클래스를 통해 **객체 간 책임 분리**와 **has-a 관계**를 학습함.
- `CarRacer`는 `private final Car myCar = new Car();` 로 자신의 차를 소유하며, 사용자의 요청을 받으면 자기 차에게 메시지를 전달(위임)하는 구조임.
- 사용자(Application) → 카레이서(요청 수신) → 자동차(실제 동작 수행) 으로 이어지는 **메시지 전송 흐름**을 구현함.
- `Car` 클래스 내부에서 `isOn`, `speed` 같은 상태값을 `private`로 캡슐화하고, 동작 메서드(`startUp`, `go`, `stop`, `turnOff`)에서 상태를 검증한 뒤 변경하도록 작성함.

#### 2. DTO(Data Transfer Object) 패턴 — `section03.dto`
- `MemberDTO` 클래스를 통해 데이터 전달 전용 객체의 표준 구조를 학습함.
- 모든 필드를 `private`으로 선언하고, `getter / setter` 메서드로만 접근하도록 캡슐화함.
- `boolean` 필드는 `getXxx()`가 아닌 `isXxx()` 형태의 getter를 사용한다는 자바 빈즈(JavaBeans) 명명 규칙을 확인함.

#### 3. 생성자(Constructor) — `section04.constructor`
- 생성자 작성 규칙 정리:
    1. 클래스 이름과 동일해야 함
    2. 반환 타입을 작성하지 않음
- `User` 클래스에 **기본 생성자**, **일부 필드 초기화 생성자**, **전체 필드 초기화 생성자**를 작성하며 오버로딩된 생성자 사용을 학습함.
- `this.필드명` — 생성된 객체 자기 자신의 필드를 가리킴.
- `this(...)` — 같은 클래스의 다른 생성자 호출. **반드시 생성자 첫 줄에 작성**해야 하며, 중복 코드를 줄이는 데 활용됨.

#### 4. 메서드 오버로딩(Overloading) — `section05.overloading`
- `BadCalculator`처럼 `addTwoInts`, `addTwoDoubles`, `addThreeInts` 와 같이 메서드명을 다르게 두는 방식의 문제점을 확인하고, `Calculator`에서 모두 `add()` 하나로 통일하는 개선안을 비교함.
- **메서드 시그니처 = 메서드 이름 + 매개변수의 (타입, 개수, 순서)**
- 오버로딩 성립 조건: 시그니처가 달라야 함.
- 오버로딩이 **성립하지 않는 경우**:
    - 반환 타입만 다른 경우
    - 접근 제한자만 다른 경우
    - 매개변수 이름만 다른 경우

#### 5. 매개변수 전달 방식 — `section05.parameter`
- **기본 자료형(primitive)**: 값(value)이 복사되어 전달 → 메서드 내부에서 변경해도 원본 불변.
- **배열, 객체(참조 자료형)**: 주소값(reference)이 복사되어 전달 → 메서드 내부에서 내용 변경 시 **원본도 함께 변경됨**.
    - `int[] arr` 의 요소 변경, `Rectangle` 객체의 `setWidth()` 호출 결과로 원본 상태가 바뀌는 것을 직접 확인함.
- **가변 인자(varargs)**: `String... toppings` 형태로 매개변수 개수를 유연하게 받을 수 있음. 내부적으로 배열로 처리됨.
    - `orderPizza("홍길동")`, `orderPizza("이순신", "치즈", "페퍼로니", "올리브")` 처럼 인자 개수를 자유롭게 호출 가능함.

### 느낀 내용

- 평소 백엔드 작업에서 자연스럽게 써왔던 DTO 패턴, 생성자 오버로딩, getter/setter 구조가 **왜 그렇게 정해졌는지** 다시 한번 정리할 수 있는 시간이었음. 특히 `boolean` 필드의 getter가 `is`로 시작하는 규칙은 Lombok이 자동 생성해주던 부분이라 무심코 넘어갔던 부분이었는데, 이번에 명확히 인지함.
- `this()` 를 활용해 생성자 간 호출 체인을 만드는 부분은 실무에서 빌더 패턴이나 정적 팩토리 메서드에 가려져 잘 안 쓰던 패턴이었지만, 가장 기본적인 코드 중복 제거 기법이라는 점에서 다시 짚어볼 만했음.
- 매개변수 전달 방식에서 **참조 자료형은 주소값이 복사된다**는 개념은 단순해 보이지만, 메서드 내부에서 의도치 않게 원본 객체를 변형시키는 사이드 이펙트의 근원이라는 점에서 매우 중요하게 느껴짐. JPA의 영속성 컨텍스트에서 dirty checking과도 맞닿아 있는 개념이라 흥미로웠음.

### 활용 방안

- **DTO 패턴**: Spring Boot에서 Request/Response DTO를 작성할 때 캡슐화 원칙을 명확히 지켜 작성함. 불변성이 필요한 경우 setter를 제거하고 생성자만 두는 방식으로 응용 가능함.
- **생성자 오버로딩 + `this()` 체인**: 엔티티 클래스에서 필수 필드만 받는 생성자와 모든 필드를 받는 생성자를 분리해 정의할 때 활용함. 향후 `@Builder` 패턴을 직접 구현할 때도 동일한 원리를 적용함.
- **메서드 오버로딩**: Service 레이어에서 검색 조건이 다른 같은 이름의 조회 메서드(`findByX`, `findByXAndY`)를 정리할 때 가독성을 높이는 도구로 사용함.
- **참조형 매개변수 주의점**: 컬렉션이나 엔티티를 메서드 인자로 넘길 때 의도치 않은 원본 변경을 방지하기 위해 방어적 복사(defensive copy)나 `Collections.unmodifiableList()` 적용 필요성을 항상 검토함.
- **가변 인자**: 로깅 유틸리티나 동적 쿼리 파라미터를 처리하는 메서드를 만들 때 호출부의 가독성을 높이는 데 활용 가능함.
