### 학습 내용

#### 1. 데이터베이스 - SQL 기초 조회 문법

##### 1-1. SELECT - 데이터 조회의 기본 (`01_select.sql`)
`SELECT 컬럼명 FROM 테이블명;` 형태로 특정 테이블에서 원하는 데이터를 조회한다.

```sql
-- 단일 컬럼
SELECT menu_name FROM tbl_menu;

-- 여러 컬럼
SELECT menu_code, menu_name, menu_price FROM tbl_menu;

-- 모든 컬럼 (* : 와일드카드)
SELECT * FROM tbl_menu;
```

- `FROM` 절 없이 `SELECT 6 + 3;` 같은 단순 연산이나 `NOW()`, `CONCAT()` 같은 내장 함수도 단독 실행 가능.
- 컬럼 별칭(Alias): `AS name`, 공백이 들어가면 `AS 'Full name'`처럼 따옴표로 감싼다.

##### 1-2. ORDER BY - 결과 정렬 (`02_orderby.sql`)
`SELECT` 문 가장 마지막에 위치하며 결과 집합을 정렬한다.

- `ASC`(오름차순, 생략 가능) / `DESC`(내림차순, **생략 불가**).
- 정렬 기준을 여러 개 줄 수 있다 → 1차 기준이 같을 때 2차 기준 적용.
- 연산 결과나 별칭으로도 정렬 가능.

```sql
-- FIELD(컬럼, 값1, 값2, ...) : 값이 목록의 몇 번째인지 숫자로 반환 → 커스텀 정렬
SELECT menu_name, orderable_status
FROM tbl_menu
ORDER BY FIELD(orderable_status, 'N', 'Y');  -- 'N'을 1순위로

-- NULL 정렬 제어 : NULL이면 1, 아니면 0
ORDER BY ref_category_code IS NULL;
```

##### 1-3. WHERE - 조건 필터링 (`03_where.sql`)
조건이 `true`인 행만 결과에 포함한다.

- 비교 연산자: `=`, `<>`, `>`, `<=` 등.
- 논리 연산자: `AND`, `OR`로 조건 조합.
- `BETWEEN A AND B`: A 이상 B 이하 (부정은 `NOT BETWEEN`).
- `LIKE`: 문자열 패턴 검색, `%`는 0개 이상의 모든 문자 와일드카드.
    - `'%마늘%'`(포함) / `'마늘%'`(시작) / `'%마늘'`(끝)
- `IN (값1, 값2, ...)`: 여러 값 중 하나와 일치하는지 검사.
- `IS NULL` / `IS NOT NULL`: NULL은 '0'이나 '빈 문자열'이 아닌 **'값이 존재하지 않는 상태'**이므로 `= NULL`로 비교할 수 없다.

##### 1-4. DISTINCT - 중복 제거 (`04_distinct.sql`)
중복을 제거하고 유니크한 값 목록만 조회. 특정 컬럼에 어떤 값이 '종류별로' 있는지 확인할 때 유용.

```sql
-- 컬럼이 여러 개면 모든 컬럼의 조합이 같을 때만 중복으로 간주
SELECT DISTINCT category_code, orderable_status FROM tbl_menu;
```

##### 1-5. LIMIT - 결과 개수 제한 (`05_limit.sql`)
결과 중 원하는 만큼만 잘라서 조회한다.

```sql
LIMIT 5;        -- 5개만
LIMIT 1, 4;     -- 1개 건너뛰고 4개 (게시판 페이징의 핵심 원리)
```

##### 1-6. JOIN - 다중 테이블 조회 (`06_join.sql`)
테이블 별칭은 JOIN에서 사실상 필수.

- **INNER JOIN**: 교집합. 양쪽에 짝이 맞는 행만 조회 (`INNER` 생략 가능).
- **LEFT / RIGHT JOIN**: 기준 테이블 데이터는 모두 보여주고, 반대쪽에 짝이 없으면 `NULL`로 채운다.
- **SELF JOIN**: 같은 테이블을 별칭 두 개로 조인 → 계층 구조(상위/하위 카테고리) 표현.

```sql
-- SELF JOIN : 하위 카테고리(a)와 상위 카테고리(b)
SELECT a.category_name AS '하위', b.category_name AS '상위'
FROM tbl_category a
JOIN tbl_category b ON a.ref_category_code = b.category_code;
```

##### 1-7. GROUP BY - 그룹화와 집계 (`07_groupby.sql`)
지정한 컬럼 값이 같은 행들을 하나의 그룹으로 묶는다.

- 집계 함수: `COUNT()`, `SUM()`, `AVG()` 등.
- `GROUP BY` 사용 시 SELECT 절에는 **GROUP BY에 사용된 컬럼**과 **집계 함수**만 올 수 있다.
- `HAVING`: 그룹화된 결과에 대한 조건. `WHERE`는 그룹화 **전** 원본 데이터 조건, `HAVING`은 그룹화 **후** 조건.
- `WITH ROLLUP`: 그룹별 집계와 함께 전체 총계 행을 추가로 보여준다.

| 작성 순서 |
| --- |
| `FROM` → `WHERE` → `GROUP BY` → `HAVING` → `ORDER BY` |

### 느낀 점 / 회고

과제 Q5-4의 다중 조인은 처음에 "테이블 2개니까 조인도 2번"이라고 생각했다가 막혔던 부분이다. 관리자도 결국 한 명의 직원이고, 직원에게 부서명을 붙이려 `DEPARTMENT`를 조인했다면 관리자에게 부서명을 붙이는 데도 똑같이 조인이 한 번 더 필요하다는 걸 깨달았다. 출력할 컬럼을 하나씩 짚으면서 "이 값은 어느 테이블, 어느 역할에서 오는가"를 먼저 정리한 뒤 FROM 절을 설계하는 습관이 필요하겠다. QueryDSL에서도 같은 엔티티를 별칭 두 개로 조인하는 상황이 나올 텐데, 오늘 SQL 레벨에서 그 구조를 손으로 짜본 게 도움이 될 것 같다.

`GROUP BY`의 SELECT 절 제약(그룹 컬럼 + 집계 함수만 허용)이나 SQL 작성 순서(`FROM → WHERE → GROUP BY → HAVING → ORDER BY`)는 JPQL이나 QueryDSL을 쓸 때도 그대로 적용되는 원칙이라, 프레임워크에 가려져 있던 기본기를 다시 정리해두는 의미가 있었다. AI Agent 쪽 RAG 작업을 하더라도 결국 메타데이터 필터링이나 집계는 SQL로 내려가는 경우가 많으니, 이 기초 문법을 정확히 잡고 가는 게 맞다.
