### 학습 내용

##### 1. 계층형 아키텍처 - 회원 관리 프로그램 (`section02.uses`)
`MemberRepository`는 `static` 배열을 사용해서 **프로그램 전체에서 하나의 저장소를 공유**하는 형태로 구현했다. Spring Boot에서 DB로 대체될 자리에 단순 배열이 들어가 있는 셈. 나중에 `JpaRepository` 인터페이스를 배울 때 이 구조가 그대로 매핑된다.

```java
public class MemberRepository {
    private final static Member[] members;
    private static int count;

    static { members = new Member[10]; }

    public static boolean store(Member[] newMembers) { ... }
    public static Member[] findAllMembers() { ... }
}
```

플래그 변수로 검색 결과 유무를 판단하는 패턴(`boolean isFound`)도 익혔다.

#### 2. Chapter 08 - 상속 (Inheritance)

##### 2-1. extends 키워드와 상속의 본질 (`section01.extend`)
`Car`를 부모로 두고 `FireCar`, `RacingCar`가 상속받는 예제.

- **상속은 코드 재사용 + 확장이다.** 부모의 필드/메서드를 그대로 물려받고, 자식만의 기능(`sprayWater()`)을 추가할 수 있다.
- **자식 객체 생성 시 부모의 생성자가 먼저 호출된다.** 출력 결과를 보면:
    ```
    Car 클래스의 기본 생성자 호출...     ← 부모 먼저
    FireCar의 기본 생성자 호출           ← 자식 나중
    ```
- 컴파일러가 자식 생성자 첫 줄에 `super();`를 자동으로 추가해준다.

##### 2-2. 접근 제한자 `protected`
```java
protected boolean isRunning() { ... }  // Car
```
- `private`: 본인만
- `default`: 같은 패키지
- **`protected`: 같은 패키지 + 상속받은 자식 클래스** (다른 패키지여도 자식이면 접근 가능)
- `public`: 어디서든

##### 2-3. super 키워드 (`section02.superkeyword`)
`Product` ← `Computer`, `SmartPhone` 구조에서 super의 두 가지 용법을 학습.

**(1) `super(...)` - 부모 생성자 호출**
```java
public Computer(String code, String brand, ..., String cpu, int ram) {
    super(code, brand, name, price, manufacturingDate);  // 반드시 첫 줄!
    this.cpu = cpu;
    this.ram = ram;
}
```
부모가 먼저 자기 필드를 초기화한 뒤, 자식이 자기 고유 필드를 초기화하는 순서가 자연스럽기 때문에 **첫 줄에 와야 한다**.

**(2) `super.메서드()` - 부모 메서드 호출**
```java
@Override
public String getInformation() {
    String parentInfo = super.getInformation();  // 부모의 정보 먼저 가져오고
    String childInfo = ", Computer [...]";       // 내 정보 덧붙이기
    return parentInfo + childInfo;
}
```
오버라이딩한 메서드 안에서 부모의 원본 동작을 재사용할 때 쓴다.

##### 2-4. 오버라이딩 규칙 (`section03.overriding`)
`VendingMachine` ← `CoffeeVendingMachine` 예제로 **무엇이 오버라이딩이 아닌지**를 정리.

| 케이스 | 오버라이딩 가능 여부 |
| --- | --- |
| 메서드 이름이 다름 | ❌ (그냥 새 메서드 추가) |
| 반환 타입이 다름 | ❌ (컴파일 에러) |
| 매개변수가 다름 | ❌ (오버로딩으로 처리됨) |
| `private` 메서드 | ❌ (자식이 존재 자체를 모름) |
| `final` 메서드 | ❌ (재정의 금지 키워드) |
| 접근 제한자를 더 좁게 (public → protected) | ❌ |
| 접근 제한자를 더 넓게 (protected → public) | ✅ |

핵심 3원칙:
1. **이름 + 반환타입 + 매개변수 목록**이 부모와 완전히 동일
2. 접근제한자는 부모와 같거나 더 넓게만
3. `private`, `final`은 절대 못 건드림

#### 3. Chapter 09 - 다형성 (Polymorphism)

##### 3-1. 업캐스팅과 동적 바인딩 (`section01.polymorphism`)
**다형성 = 하나의 타입(부모)으로 여러 형태(자식)를 다룰 수 있는 능력.**

다형성이 없을 때는 사육사가 동물 종류마다 별도의 메서드를 가져야 한다:
```java
public void feed(Cat cat) { ... }
public void feed(Tiger tiger) { ... }
// 동물 100종이면 메서드 100개...
```

다형성을 적용하면:
```java
public void feed(Animal animal) {
    animal.eat();  // 동적 바인딩
}
```
한 줄로 끝난다. 새 동물이 추가되어도 `feed` 메서드는 수정 안 해도 됨.

**핵심 개념 2가지:**
- **업캐스팅**: 자식 객체를 부모 타입으로 가리키기 → `Animal a = new Cat();` (자동, 묵시적)
- **동적 바인딩**: 컴파일 시점에는 `Animal.cry()`로 보이지만, **런타임에 실제 객체(Cat)의 오버라이딩된 메서드가 실행됨**

##### 3-2. 다운캐스팅과 instanceof
업캐스팅된 변수에서는 자식 고유 메서드(`Cat.jump()`)에 접근할 수 없다. 필요하면 **다운캐스팅**:
```java
Animal a1 = new Cat();
((Cat) a1).jump();         // OK
((Cat) a2).jump();         // Tiger를 Cat으로? → ClassCastException (런타임 에러)
```

다운캐스팅 전에 **`instanceof`로 안전 확인**하는 게 정석:
```java
if (a1 instanceof Cat) {
    ((Cat) a1).jump();
}
```

##### 3-3. 추상 클래스 (`section02.abstractclass`)
일반 메서드만 있으면 자식이 오버라이딩을 **'깜빡할 수 있다'**는 게 문제. 추상 클래스는 자식이 반드시 구현하도록 **강제**한다.

```java
public abstract class Player {
    public abstract void attack();      // 몸통이 없음 → 자식이 강제 구현
    public void levelUp() { ... }       // 일반 메서드도 가능 (공통 동작)
}
```
- `abstract class`는 직접 `new`로 생성 불가 (미완성 설계도이기 때문)
- 추상 메서드를 가진 클래스는 반드시 `abstract`로 선언해야 함
- 단일 상속만 가능 (`extends`는 하나만)

##### 3-4. 인터페이스 (`section03.interfaceimplements`)
`IConnectable` 인터페이스로 USB 같은 **표준 규격**을 정의하고, `Keyboard`/`Mouse`가 이를 구현. `Computer`는 구현체가 누구든 상관없이 인터페이스 타입으로 받는다.

```java
public interface IConnectable {
    int MAX_CONNECTION_COUNT = 4;       // 자동 public static final
    void connect();                      // 자동 public abstract
    void disconnect();

    default void showStatus() {          // default 메서드는 기본 구현 제공
        System.out.println("장치가 대기 상태입니다.");
    }
}
```

`Computer`의 `connectDevice` 파라미터 타입이 `IConnectable`인 게 다형성의 정수다. 새로운 장치(USB, 모니터 등)가 추가되어도 `Computer` 코드는 손댈 필요가 없다 → **OCP(개방-폐쇄 원칙)**.

##### 3-5. 추상 클래스 vs 인터페이스 (오늘의 정리표)

| 구분 | 추상 클래스 | 인터페이스 |
| --- | --- | --- |
| 키워드 | `abstract class` | `interface` |
| 사용 | `extends` | `implements` |
| 다중 상속 | 단일 상속만 | 여러 개 구현 가능 |
| 일반 메서드 | 가능 | `default` 메서드로만 |
| 변수 | 일반 필드 가능 | 상수만 가능 |
| 목적 | 공통 기능 + 강제 | 순수 규격/계약 정의 |

### 느낀 점 / 회고

`chap07`의 회원 관리 프로그램에서 Controller-Service-Repository 구조를 직접 손으로 짜본 게 가장 인상 깊었다. 작년 인턴 때 Spring Boot로 작업했던 `Estimate`, `Customer` 도메인의 3계층 구조가 결국 이 단순한 `Member[]` 배열 저장소에서 출발한 거라는 게 명확히 이해됐다. `MemberRepository`의 `static` 배열이 나중에 `JpaRepository`로, 그 뒤에 PostgreSQL로 대체되는 흐름이 머릿속에서 자연스럽게 그려진다.

`chap08-09`는 OOP의 본질을 압축한 단원이었다. 특히 `Zookeeper.feed(Animal animal)` 한 줄로 모든 동물을 처리하는 코드를 보면서, **확장에는 열려있고 수정에는 닫혀있다**는 OCP 원칙이 다형성 위에서 어떻게 작동하는지 체감했다. Spring의 DI(의존성 주입)도 결국 인터페이스 타입으로 받아서 런타임에 구현체를 갈아끼우는 패턴이라는 점에서 오늘 배운 내용이 그대로 이어진다.

오버라이딩 규칙은 IntelliJ의 `@Override` 체크 덕에 대부분 자동으로 잡히지만, **왜 그 규칙이 그래야 하는지**(접근제한자가 좁아지면 자식이 부모 자리를 대체할 수 없으니까)를 정확히 이해하고 가는 게 중요하겠다.
