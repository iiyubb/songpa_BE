#### 1. Chapter 10 - 자바 표준 API

##### 1-1. Object 클래스 - toString / equals / hashCode (`section01.object`)
모든 클래스의 최상위 부모인 `Object`. 별도 상속 표시가 없어도 모든 클래스는 자동으로 `Object`를 상속한다. 그래서 어떤 객체든 `toString()`, `equals()`, `hashCode()`를 가지고 있고, 의미 있게 쓰려면 **오버라이딩이 필수**.

`Book` 클래스 예제로 세 메서드를 직접 재정의했다.

```java
@Override
public String toString() {
    return "Book {" + "number=" + number + ", title=" + title + ", ...}";
}

@Override
public boolean equals(Object o) {
    if (o == null || getClass() != o.getClass()) return false;
    Book book = (Book) o;
    return number == book.number && price == book.price
        && Objects.equals(title, book.title)
        && Objects.equals(author, book.author);
}

@Override
public int hashCode() {
    return Objects.hash(number, title, author, price);
}
```

핵심 정리:
- **`toString()`**: 기본은 `클래스명@해시코드` → 디버깅/로깅용으로 의미 있는 문자열로 재정의.
- **`==` vs `equals()`**: `==`는 주소값 비교(동일성), `equals()`는 내용 비교(동등성). 같은 값을 담은 두 객체를 "같다"고 판단하려면 오버라이딩 필요.
- **equals-hashCode 규약**: `equals()`가 true면 `hashCode()`도 반드시 같아야 한다. 안 그러면 `HashSet`/`HashMap`이 객체를 못 찾음.

> 해시 컬렉션 동작 순서: ① `hashCode()`로 버킷 위치 찾기 → ② 같은 버킷 안에서 `equals()`로 동등성 비교. hashCode가 다르면 애초에 다른 버킷을 보므로 equals까지 도달조차 못 한다.

##### 1-2. String 클래스 (`section01.string`)
상품 데이터 파싱 예제로 주요 메서드를 정리.

| 메서드 | 역할 |
| --- | --- |
| `trim()` | 앞뒤 공백 제거 |
| `toUpperCase() / toLowerCase()` | 대소문자 변환 (검색·비교 시 통일용) |
| `indexOf(char)` | 문자 위치 반환, **없으면 -1** |
| `substring(start, end)` | 구간 추출 (end 미포함) |
| `replace(old, new)` | 치환 |
| `split(regex)` | 구분자로 분리, 배열 반환 |

**String 불변성**:
```java
String str1 = "java";
String str2 = "java";
String str3 = new String("java");
String str4 = new String("java");

System.out.println(str1 == str2);      // true  (String Pool 공유)
System.out.println(str3 == str4);      // false (Heap에 각각 생성)
System.out.println(str3.equals(str4)); // true
```

리터럴로 만든 문자열은 String Pool에서 공유되지만 `new String(...)`은 Heap에 매번 새 객체를 만든다. `str2 += "mysql"`을 하면 str2가 가리키는 주소 자체가 바뀌는 거지 원본 "java"가 변경되는 게 아니다(불변).

##### 1-3. StringBuilder - 문자열 누적의 정답 (`section01.stringbuilder`)
String의 불변성 때문에 `+=` 연산은 매번 새 객체를 만든다. 50,000번 누적해서 비교했더니:

```
String합치기 5만번:        ~ 수 초 단위
StringBuilder합치기 5만번:  ~ 수 ms 단위
```

차이가 **수백 배 이상**. 반복문 안에서 문자열을 더해야 한다면 무조건 `StringBuilder`를 써야 한다.

```java
StringBuilder sb = new StringBuilder("javamysql");
sb.delete(4, 9);        // "java"
sb.insert(0, "my");     // "myjava"
sb.reverse();           // "avajym"
String result = sb.toString();
```

가변(mutable) 객체이므로 메서드들이 객체 자체를 수정하고 자기 자신을 반환 → 메서드 체이닝 가능.

> 실무 메모: 멀티스레드 환경에서는 동기화된 `StringBuffer`를 쓰지만, 일반적으로는 `StringBuilder`가 빠르고 충분하다.

##### 1-4. java.time 패키지 (`section01.time`)
JDK 8부터 도입된 **불변(immutable) + 스레드 안전** 날짜/시간 API. 기존 `Date`, `Calendar`의 문제를 해결한 표준.

```java
LocalDate departureDate = LocalDate.of(2026, 12, 25);
LocalTime departureTime = LocalTime.of(13, 8);
LocalDateTime trainDeparture = LocalDateTime.of(departureDate, departureTime);

trainDeparture.minusMinutes(20);   // 새 객체 반환 (원본 불변)
trainDeparture.isBefore(LocalDateTime.now());

DateTimeFormatter formatter = DateTimeFormatter.ofPattern("yyyy년 MM월 dd일 HH시 mm분 출발");
String formattedDate = trainDeparture.format(formatter);

// 문자열 → 날짜 객체로 파싱
LocalDate parsedChristmas = LocalDate.parse("261225",
    DateTimeFormatter.ofPattern("yyMMdd"));
```

세 종류의 차이:
- `LocalDate`: 날짜만 (`2026-12-25`)
- `LocalTime`: 시간만 (`13:08:00`)
- `LocalDateTime`: 둘 다 (`2026-12-25T13:08:00`)

##### 1-5. Wrapper 클래스 (`section01.wrapper`)
기본형(`int`, `double`)을 객체로 포장한 게 Wrapper(`Integer`, `Double`).

```java
String heightStr = "170.5";
double height = Double.parseDouble(heightStr);    // 파싱: 문자열 → 기본형

Integer age = 10;       // 오토 박싱 (Integer.valueOf(10))
int unboxedAge = age;   // 오토 언박싱 (age.intValue())
```

**제네릭/컬렉션에 기본형 못 쓰는 이유**: `List<int>` 안 됨, `List<Integer>` 써야 함. 제네릭은 내부적으로 Object로 다루기 때문에 객체 타입(참조형)만 허용.

**값 비교 함정**:
```java
Integer num3 = 128;
Integer num4 = 128;
System.out.println(num3 == num4);       // false (객체 주소 비교)
System.out.println(num3.equals(num4));  // true
```
Wrapper는 객체이므로 값 비교는 반드시 `equals()`. (-128 ~ 127 범위는 캐싱돼서 `==`도 true가 나오는데, 이건 함정이라 절대 의존하면 안 됨.)

#### 2. Chapter 11 - 제네릭 (Generics)

##### 2-1. 제네릭이 필요한 이유 (`section01.generic`)
`Object`로 모든 타입을 받는 `NormalBox` 방식의 문제:

```java
NormalBox normalBox = new NormalBox();
normalBox.setContent("안녕하세요");
String content = (String) normalBox.getContent();  // 꺼낼 때 강제 형변환

normalBox.setContent(123);  // 실수로 숫자 넣어도 컴파일 에러 안 남
String wrongContent = (String) normalBox.getContent();  // 런타임 ClassCastException!
```

`Object` 박스는 **타입 안전성이 없다**. 컴파일러가 잡아줄 기회를 잃고 런타임에 터진다.

제네릭을 적용한 `GenericBox<T>`:

```java
public class GenericBox<T> {
    private T content;
    public T getContent() { return content; }
    public void setContent(T content) { this.content = content; }
}
```

```java
GenericBox<String> stringBox = new GenericBox<>();
stringBox.setContent("hello world");
String strContent = stringBox.getContent();  // 형변환 불필요!

GenericBox<Integer> integerBox = new GenericBox<>();
integerBox.setContent(100);
// integerBox.setContent("string");  // 컴파일 에러로 미리 막힘
```

**제네릭의 장점**:
1. 형변환 불필요 (꺼낼 때 이미 타입이 정해져 있음)
2. **컴파일 시점에 타입 체크** → 런타임 에러를 컴파일 에러로 끌어올림
3. 코드 재사용 (타입마다 클래스 따로 만들 필요 없음)

> 제약: 타입 파라미터 자리에는 **참조형만** 가능. `GenericBox<int>` 안 되고 `GenericBox<Integer>` 써야 함.

##### 2-2. 제한된 타입 파라미터 (`section02.extend`)
`<T>` 자리에 아무 타입이나 들어오면 곤란할 때 사용.

```java
public class RabbitFarm<T extends Rabbit> {
    private T animal;
    // ...
}
```

`T`는 `Rabbit` 또는 그 자손만 허용. 클래스 계층:
```
Animal(interface) ← Mammal ← Rabbit ← Bunny ← DrunkenBunny
                   ← Reptile ← Snake
```

```java
RabbitFarm<Rabbit> farm1 = new RabbitFarm<>(new Rabbit());
RabbitFarm<Bunny> farm2 = new RabbitFarm<>(new Bunny());
RabbitFarm<DrunkenBunny> farm3 = new RabbitFarm<>(new DrunkenBunny());

// RabbitFarm<Snake> farm = ...; // 컴파일 에러: Snake는 Rabbit 자손이 아님
```

`extends`라는 키워드를 썼지만 인터페이스 구현체에도 동일하게 사용. 클래스 상속이든 인터페이스 구현이든 제네릭 제한에서는 `extends`로 통일.

##### 2-3. 와일드카드 - `?`, `? extends`, `? super`
**메서드 파라미터에서 다양한 타입의 제네릭을 받아들이고 싶을 때** 사용. `T`와 와일드카드는 쓰는 자리가 다르다.

```java
public class FramManager {

    // ① 무제한 와일드카드: 어떤 RabbitFarm이든 다 받음
    public void managerAnyFarm(RabbitFarm<?> farm) {
        farm.getAnimal().cry();
    }

    // ② 상한 경계: Bunny 또는 그 자손만
    public void managerBunnyFarm(RabbitFarm<? extends Bunny> farm) {
        farm.getAnimal().cry();
    }

    // ③ 하한 경계: Bunny 또는 그 조상만
    public void managerRabbitOrBunnyFarm(RabbitFarm<? super Bunny> farm) {
        farm.getAnimal().cry();
    }
}
```

테스트:
```java
farmManager.managerAnyFarm(farm1);    // OK
farmManager.managerAnyFarm(farm2);    // OK
farmManager.managerAnyFarm(farm3);    // OK

// farmManager.managerBunnyFarm(farm1);  // 컴파일 에러: Rabbit은 Bunny 자손 아님
farmManager.managerBunnyFarm(farm2);     // OK (Bunny)
farmManager.managerBunnyFarm(farm3);     // OK (DrunkenBunny)
```

| 표기 | 의미 | 흔히 쓰는 상황 |
| --- | --- | --- |
| `<?>` | 모든 타입 | 읽기만 할 때, 타입 무관 |
| `<? extends T>` | T 또는 자손 | T 계열에서 **꺼낼(읽을)** 때 |
| `<? super T>` | T 또는 조상 | T 또는 그 자손을 **넣을(쓸)** 때 |

영어로 **PECS** 원칙: **P**roducer-**E**xtends, **C**onsumer-**S**uper.

### 느낀 점 / 회고

`Object`의 equals/hashCode 규약은 작년 Findex 프로젝트에서 JPA Entity에 `@EqualsAndHashCode(of = "id")`를 Lombok으로 붙였던 게 떠올랐다. 그때는 그냥 관례처럼 붙였는데, 오늘 `HashSet`이 `hashCode → equals` 순서로 객체를 찾는 메커니즘을 보고 나니 **왜 영속성 컨텍스트의 1차 캐시가 `@Id` 기반 equals를 요구하는지** 명확해졌다. 두 객체가 같은 엔티티인지 판단하는 기준이 곧 PK라는 거니까.

String의 불변성 + StringBuilder 성능 차이는 `대모산` 인턴 때 5천 건짜리 견적 Excel을 파싱하면서 느꼈던 거랑 정확히 일치한다. 그때 `String` 누적으로 PDF 본문을 만들다가 응답이 느려져서 결국 `StringBuilder`로 갈아엎었던 기억이 있는데, 오늘 5만 번 벤치마크 결과를 직접 찍어보니 그때 왜 그렇게 느렸는지 수치로 확인됐다.

제네릭은 솔직히 `chap07-object-array`에서 `Object[]`로 만든 회원 저장소를 떠올리면 답이 너무 명확하다. 그 시점에 누군가 실수로 `Member`가 아닌 `String`을 넣었어도 컴파일러는 가만히 있었을 거다. `List<Member>`처럼 제네릭으로 묶는 순간 그런 실수 자체가 컴파일 에러로 막힌다 — **런타임 버그를 컴파일 시점으로 끌어올린다**는 게 정적 타입 언어의 핵심 가치라는 걸 새삼 느꼈다.

와일드카드 `? extends` / `? super`는 처음 봤을 땐 추상적이었는데, `RabbitFarm` 예제로 직접 돌려보니 **"메서드 입장에서 인자로 어떤 범위까지 허용할지를 정하는 문법"**이라는 감이 잡혔다. PECS 원칙(생산자는 extends, 소비자는 super)은 컬렉션 API를 깊게 쓸 때 다시 만나게 될 거라 메모해두자. Spring에서도 `BeanFactory.getBeansOfType(Class<? extends T>)` 같은 시그니처를 자주 보게 되는데, 오늘 이후로는 그 의미가 보일 것 같다.
