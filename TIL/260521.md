#### 1. Chapter 03 - Servlet에서 JSON 응답하기

##### 1-1. Servlet + JSON API 구조 (`chap03-1_json-test`)
HTML 폼 기반의 전통적인 Servlet 응답 방식에서 벗어나, **JSON으로 통신하는 REST 스타일 API**를 직접 만들어봤다. 모던 백엔드의 기본 구조다.

```
브라우저(index.html)
    ↓ fetch JSON
@WebServlet("/api/memos") MemoApiServlet
    ↓
static List<MemoDTO>  ← in-memory 저장
```

핵심은 **Jackson 라이브러리**의 `ObjectMapper`. 자바 객체와 JSON 사이의 변환을 담당한다.

```java
@WebServlet("/api/memos")
public class MemoApiServlet extends HttpServlet {

    private static final ObjectMapper mapper = new ObjectMapper();
    private static final List<MemoDTO> memos = new ArrayList<>();

    @Override
    protected void doGet(HttpServletRequest req, HttpServletResponse resp) throws IOException {
        resp.setContentType("application/json; charset=UTF-8");
        // 자바 객체 → JSON으로 직렬화해서 응답 본문에 출력
        mapper.writeValue(resp.getWriter(), memos);
    }

    @Override
    protected void doPost(HttpServletRequest req, HttpServletResponse resp) throws IOException {
        // JSON → 자바 객체로 역직렬화
        MemoDTO requestMemo = mapper.readValue(req.getReader(), MemoDTO.class);
        // ...
    }
}
```

Jackson 의존성 (Gradle):
```gradle
implementation 'tools.jackson.core:jackson-databind:3.1.2'
```

##### 1-2. DTO 설계 규칙
Jackson이 JSON ↔ 자바 객체 변환을 하려면 DTO에 반드시 두 가지가 필요하다.

```java
public class MemoDTO {
    private int id;
    private String content;

    public MemoDTO() {}                        // ① 기본 생성자 (역직렬화용)

    public MemoDTO(int id, String content) {   // 코드에서 객체 생성용
        this.id = id;
        this.content = content;
    }

    // ② getter/setter (Jackson이 필드값을 읽고 쓰는 데 사용)
    public int getId() { return id; }
    public void setId(int id) { this.id = id; }
    public String getContent() { return content; }
    public void setContent(String content) { this.content = content; }
}
```

**기본 생성자가 핵심**. Jackson은 `new MemoDTO()`로 빈 객체를 만든 뒤 setter로 값을 채우는 방식인데, 자바는 다른 생성자를 직접 정의하는 순간 **기본 생성자를 자동 생성하지 않는다**. 매개변수 있는 생성자만 두면 Jackson이 객체를 못 만들어서 `InvalidDefinitionException`이 터진다.

> 필드명과 JSON key 이름은 일치해야 한다. 다르게 하려면 `@JsonProperty("xxx")` 어노테이션 필요.

##### 1-3. HTTP 상태 코드와 에러 응답
Servlet에서 응답 코드를 직접 제어한다.

```java
if (content.isEmpty()) {
    resp.setStatus(HttpServletResponse.SC_BAD_REQUEST);   // 400
    mapper.writeValue(resp.getWriter(), new ErrorResponse("content is required"));
    return;
}

MemoDTO savedMemo = new MemoDTO(sequence++, content);
memos.add(savedMemo);
resp.setStatus(HttpServletResponse.SC_CREATED);           // 201
mapper.writeValue(resp.getWriter(), savedMemo);
```

에러 응답도 JSON 형태로 통일하기 위해 `ErrorResponse`라는 작은 DTO를 따로 만들었다. 클라이언트 입장에서는 성공이든 실패든 항상 JSON으로 받게 되니 처리가 깔끔하다.

#### 2. Chapter 03 - Servlet에 JDBC 붙이기 (`chap03-2_json-jdbc`)

##### 2-1. in-memory에서 DB로
chap03-1의 `static List`를 **MySQL DB**로 교체했다. Servlet/JSON 레이어는 그대로 두고 데이터 저장소만 JDBC로 바꾸는 작업.

```
Next.js (localhost:3000)
    ↓ fetch /api/memos
[next.config.mjs rewrites] → Tomcat:8080
    ↓
MemoApiServlet
    ↓
MemoService → MemoDao
    ↓ JDBC
MySQL (memo_db.tbl_memo)
```

DB 스키마:
```sql
CREATE DATABASE IF NOT EXISTS memo_db;
USE memo_db;

CREATE TABLE tbl_memo (
    memo_id    INT AUTO_INCREMENT PRIMARY KEY,
    content    VARCHAR(255) NOT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

##### 2-2. 계층 분리 (Servlet / Service / DAO)
하나의 Servlet에 SQL까지 다 때려박을 수도 있지만, 책임을 분리하는 게 정석이다.

| 계층 | 책임 |
| --- | --- |
| **Servlet** | HTTP 요청/응답, JSON 직렬화/역직렬화 |
| **Service** | 비즈니스 로직, Connection 수명 관리 |
| **DAO** | SQL 실행, ResultSet → 객체 매핑 |
| **JDBCTemplate** | Connection 획득/종료 유틸 |

```java
// Service는 Connection을 받아서 DAO에 넘기고, 마지막에 close 책임을 진다
public List<MemoDto> findAllMemos() {
    Connection con = getConnection();
    try {
        return memoDao.selectAllMemos(con);
    } finally {
        close(con);
    }
}
```

이 구조가 곧 Spring의 `@Controller → @Service → @Repository` 계층 분리의 원형이라는 걸 코드를 짜면서 체감했다.

##### 2-3. JDBC 4단계 패턴
JDBC는 항상 같은 흐름을 반복한다: **Connection → PreparedStatement → ResultSet → close**.

```java
public List<MemoDto> selectAllMemos(Connection con) {
    PreparedStatement pstmt = null;
    ResultSet rset = null;
    List<MemoDto> memos = new ArrayList<>();

    String query = """
            SELECT memo_id, content, created_at
            FROM tbl_memo
            ORDER BY memo_id DESC
            """;

    try {
        pstmt = con.prepareStatement(query);
        rset = pstmt.executeQuery();

        while (rset.next()) {
            memos.add(new MemoDto(
                rset.getInt("memo_id"),
                rset.getString("content"),
                formatTimestamp(rset.getTimestamp("created_at"))
            ));
        }
    } catch (SQLException e) {
        e.printStackTrace();
    } finally {
        close(rset);   // 역순으로 close
        close(pstmt);
    }
    return memos;
}
```

- **`PreparedStatement`**: `?` 자리표시자로 파라미터 바인딩 → SQL Injection 방어 + SQL 캐싱
- **자원 해제는 역순** (ResultSet → Statement → Connection). 외부 자원이라 finally에서 반드시 close.
- Java 15+ **Text Block (`"""`)** 으로 멀티라인 SQL이 훨씬 읽기 좋아졌다.

##### 2-4. INSERT 후 자동 생성된 PK 받아오기
`AUTO_INCREMENT` 컬럼이 있는 테이블은 INSERT 직후 생성된 ID를 받아와야 클라이언트에 반환할 수 있다.

```java
String query = "INSERT INTO tbl_memo(content) VALUES (?)";

// Statement.RETURN_GENERATED_KEYS : 생성된 PK를 받을 수 있도록 설정
pstmt = con.prepareStatement(query, Statement.RETURN_GENERATED_KEYS);
pstmt.setString(1, content);
pstmt.executeUpdate();

// DB가 자동 생성한 키 목록을 ResultSet 형태로 받음
generatedKeys = pstmt.getGeneratedKeys();
if (generatedKeys.next()) {
    int id = generatedKeys.getInt(1);
    return selectMemoById(con, id);   // 새로 생성된 메모를 다시 조회해서 반환
}
```

JPA의 `@GeneratedValue` 뒤에서도 결국 이 로직이 돌아간다.

##### 2-5. db.properties와 드라이버 로딩
DB 접속 정보를 코드에 박지 않고 외부 파일로 분리.

```properties
# src/main/resources/db.properties
driver=com.mysql.cj.jdbc.Driver
url=jdbc:mysql://localhost:3306/memo_db
user=ohgiraffers
password=ohgiraffers
```

```java
public static Connection getConnection() {
    Properties prop = new Properties();
    try (InputStream input = JDBCTemplate.class.getClassLoader()
                                .getResourceAsStream("db.properties")) {
        prop.load(input);
        Class.forName(prop.getProperty("driver"));
        return DriverManager.getConnection(prop.getProperty("url"), prop);
    } catch (IOException | ClassNotFoundException | SQLException e) {
        throw new RuntimeException(e);
    }
}
```

> MySQL Connector/J 8 이상은 `com.mysql.cj.jdbc.Driver`가 정식이다. 구버전 `com.mysql.jdbc.Driver`를 쓰면 deprecated 경고가 뜬다. 그리고 SPI 자동 등록 덕분에 사실 `Class.forName()` 호출 자체도 생략 가능하지만 명시적으로 두는 게 흐름을 이해하기에는 좋다.

Gradle 의존성:
```gradle
implementation 'com.mysql:mysql-connector-j:9.4.0'
```

#### 3. Next.js 프론트엔드와 Tomcat 백엔드 연동

##### 3-1. 두 개의 개발 서버
풀스택 개발 환경에서는 두 서버를 동시에 띄운다.

| 서버 | 포트 | 실행 |
| --- | --- | --- |
| Tomcat (백엔드) | 8080 | IntelliJ에서 RUN |
| Next.js (프론트) | 3000 | 터미널 `npm run dev` |

Spring Boot가 Tomcat을 내장해서 한 번에 띄워주는 것과 달리, 외장 Tomcat은 따로 관리해야 한다. 이 분리된 상태를 직접 경험해보면 Spring Boot가 어떤 편의를 자동화하고 있는지 보인다.

##### 3-2. Next.js rewrites로 CORS 회피
`localhost:3000`에서 `localhost:8080`을 호출하면 origin이 달라서 CORS에 막힌다. 백엔드에 CORS 헤더를 직접 박는 방법도 있지만, 개발 환경에서는 **Next.js의 rewrites 기능**으로 프록시하는 게 훨씬 깔끔하다.

```javascript
// next.config.mjs
const nextConfig = {
    async rewrites() {
        return [{
            source: '/api/:path*',
            destination: 'http://localhost:8080/api/:path*',
        }];
    },
};
```

이렇게 하면 프론트에서 `/api/memos`로 호출 시 Next.js가 내부적으로 `localhost:8080/api/memos`로 전달한다. 브라우저는 같은 origin으로 인식하므로 CORS가 걸리지 않는다.

##### 3-3. Zustand 상태 관리
글로벌 상태 관리 라이브러리. Redux보다 훨씬 간단하다.

```javascript
export const useMemoStore = create((set) => ({
    memos: [],
    loading: false,
    error: '',

    fetchMemos: async () => {
        set({ loading: true, error: '' });
        try {
            const memos = await getMemos();
            set({ memos });
        } catch (error) {
            set({ error: error.message });
        } finally {
            set({ loading: false });
        }
    },

    addMemo: async (content) => {
        try {
            const savedMemo = await createMemo(content);
            set((state) => ({
                memos: [...state.memos, savedMemo],   // 불변성 유지
            }));
        } catch (error) {
            set({ error: error.message });
        }
    },
}));
```

- 상태와 액션을 한 스토어에 묶을 수 있다.
- 배열 업데이트 시 `state.memos.push(...)`처럼 직접 수정하지 않고 `[...state.memos, newItem]` 형태로 **새 배열을 만드는** 게 React 불변성 원칙. `gooodock` 프로젝트에서 구독 도메인 다룰 때 익숙해진 패턴이다.

### 느낀 점 / 회고

오늘 Servlet/JDBC/JSON 실습이 왜 커리큘럼에 들어 있는지 다시 생각해봤다. Codeit 부트캠프 때부터 Spring Boot + JPA로만 작업해와서 사실 "이 옛날 걸 왜?"라는 생각이 잠깐 들었는데, 직접 짜보니 결국 `@RestController`가 `HttpServletRequest/Response`를 받아 `ObjectMapper`로 JSON 변환을 하는 거고, `JpaRepository`가 내부적으로 `Connection → PreparedStatement → ResultSet` 패턴을 돌리는 거였다. Spring이 자동화한 레이어들을 손으로 한 번 짜보니 "마법처럼 동작하는 Spring"에서 한 단계 내려와서 본 느낌이다. 면접 준비할 때 "DispatcherServlet이 뭐냐", "JPA가 내부적으로 어떻게 동작하느냐" 같은 질문은 결국 오늘 짠 이 코드를 설명할 수 있느냐를 묻는 거라는 생각이 들었다.
